# AI Job Targeting Pack

Built from live job-market sampling and cross-checked against the seven-skill model from the Nate B Jones video.

Purpose:
- define the target role family clearly
- create repeatable search queries and board filters
- rewrite positioning around what employers are actually buying

## 1. Target-job taxonomy

The market is real, but fragmented. The same underlying skill cluster appears under multiple titles.

## Tier 1 — strongest target roles

These are the best fit because they map most directly to the work already being done across Halo / Hermes / agent workflows.

### A. Applied AI Engineer

Typical signals:
- LLM-powered features from prototype to production
- orchestration workflows
- evaluation pipelines
- RAG / memory systems
- observability and debugging
- reliability, latency, and cost improvements

Why this is a fit:
- strongest overlap with actual hands-on agent workflow design
- broad enough to include systems, tooling, and product-facing work
- avoids the fluff of "prompt engineer" while still covering prompt/context work

Priority: Very high

### B. AI Systems Engineer

Typical signals:
- agent systems
- tool use
- backend integration
- tracing and monitoring
- operational robustness
- production behavior of AI systems

Why this is a fit:
- good for presenting as someone who makes AI systems behave, not just demo nicely
- fits the Halo / listen-direct / Hermes style of work

Priority: Very high

### C. LLM / Agent Engineer

Typical signals:
- prompt design
- tools and guardrails
- evaluation harnesses
- multi-agent workflows
- retrieval and context management
- sandboxed or auditable execution

Why this is a fit:
- good fit for contract and startup roles
- especially strong when the role is explicit about tools, prompts, and guardrails

Priority: Very high

### D. Evaluation / Reliability flavored AI Engineer

Typical titles:
- LLM Evaluation Engineer
- AI/LLM Evaluation & Alignment Engineer
- AI Reliability Engineer
- AI Safety / Guardrails Engineer

Typical signals:
- evaluation frameworks
- online/offline metrics
- red-teaming / adversarial testing
- classifier or rules-based enforcement
- output quality / agent behavior controls
- safety and policy enforcement

Why this is a fit:
- the market strongly values this
- gives a differentiated angle versus generic app-builders
- especially useful if portfolio projects visibly demonstrate eval and reliability discipline

Priority: High

## Tier 2 — adjacent target roles

### E. Prompt + Context Engineer

Typical signals:
- prompt strategy
- retrieval pipelines
- memory and context engineering
- context-window discipline
- multi-agent coordination
- production observability

Why this is a fit:
- real demand exists
- useful transitional label
- but should usually be framed as part of a larger systems capability, not as the whole identity

Priority: High, but best used tactically rather than as the main identity

### F. Agentic AI Product Manager / Technical Product Manager

Typical signals:
- product vision for agentic systems
- defining workflows and success metrics
- balancing safety, utility, and delivery
- coordinating engineering, product, and operations

Why this is a fit:
- plausible if framed from systems judgment rather than roadmap theatre
- especially strong if paired with evidence of operational and evaluative rigor

Priority: Medium-high

## Tier 3 — lower-priority / avoid as primary label

### G. Prompt Engineer

Problem:
- too narrow
- often interpreted as superficial or junior
- can undersell the operational systems work

Use only when:
- the role itself explicitly uses the title
- and the application reframes it as prompt + context + eval + orchestration

Priority: Low as a primary identity

### H. AI Architect

Problem:
- can drift into overly senior / enterprise-deck territory
- may suggest more ivory-tower design than working systems

Use when:
- the role explicitly calls for system design, trust boundaries, multi-agent orchestration, and governance

Priority: Medium, selectively

## Preferred identity stack

Primary label:
- Applied AI Engineer

Back-up labels depending on posting:
- AI Systems Engineer
- LLM / Agent Engineer
- Evaluation & Reliability Engineer
- Technical Product Manager, Agentic AI

## Board-specific filters

### LinkedIn
Use filters:
- Location: United Kingdom
- Remote: Remote
- Experience: Mid-Senior / Director only when appropriate
- Keywords: use one concept cluster at a time, not all at once

Best LinkedIn keyword combinations:
- "LLM evaluation" OR guardrails OR observability
- "agents" AND orchestration AND Python
- "RAG" AND evaluation AND tracing
- "context engineering" OR retrieval pipelines

### Indeed
Indeed is better for broader scraping language.
Use searches like:
- ai evaluation jobs
- ai workflow jobs
- llm rag jobs
- prompt architect jobs
- agentic product manager jobs

Then filter manually for:
- actual LLM / agent work
- explicit evaluation / retrieval / guardrail language
- not generic data-science slop

### Lever / Ashby / company pages
Best for high-signal roles.
Search directly for:
- applied ai
- llm
- ai systems
- orchestration
- evaluation
- guardrails

### CWJobs / Otta / Jobgether / Remote boards
Use these for:
- UK-specific senior engineering roles
- startup remote roles
- contract and hybrid AI-system positions

## Query logic notes

Do not search only for "AI jobs".
Search by capability clusters:
- evaluation
- guardrails
- orchestration
- retrieval
- context
- reliability
- observability
- prompt versioning

This gets closer to the real jobs.

## Negative filters / junk filters

Skip or down-rank roles that are:
- mostly generic chatbot copywriting
- pure prompt-writing without systems responsibility
- vague "AI enthusiast" jobs
- AI trainer / labeling work unless strategically useful for cashflow
- roles with no mention of evaluation, reliability, tooling, integration, or measurable outcomes

## 3. Resume rewrite oriented around market signals

This is not a full CV rewrite. It is the positioning layer and bullet-bank direction.

## Headline options

### Option 1
Applied AI Engineer building reliable agent workflows, evaluation systems, and context-aware automation.

### Option 2
Senior software engineer focused on AI systems design: orchestration, evaluation, guardrails, and retrieval-aware workflows.

### Option 3
AI systems / product-minded engineer turning LLM prototypes into governable, evaluable, production-ready workflows.

## Professional summary — draft

Senior software engineer focused on making AI systems useful in real workflows rather than merely impressive in demos. Strongest in task specification, context design, multi-step workflow decomposition, evaluation loops, and operational guardrails for LLM-driven systems. Built and iterated on agent-oriented tooling spanning orchestration, retrieval-aware documentation, structured review, cron-driven briefings, and workflow automation. Especially interested in roles at the intersection of applied AI, systems design, evaluation, and product judgment.

## Positioning bullets to reuse

### Core capability bullets
- Designed and operated AI-assisted workflows with explicit specifications, structured outputs, and review checkpoints.
- Built multi-step agent processes by decomposing ambiguous work into scoped, verifiable subtasks.
- Improved AI output reliability through evaluation loops, adversarial review patterns, and failure diagnosis.
- Created context architectures and documentation structures that improved agent retrieval quality and reduced drift.
- Applied guardrails and approval boundaries to AI-driven workflows based on blast radius, reversibility, and verifiability.
- Evaluated model and tool choices based on quality, latency, cost, and operational fit.

### Systems / backend flavored bullets
- Built backend and workflow tooling for agent-oriented systems, including job execution, orchestration surfaces, and scheduled operational pipelines.
- Designed deterministic wrappers, logging, and review mechanisms around probabilistic model behavior.
- Integrated CLI, cron, and structured data flows to make AI-assisted systems inspectable and repeatable.

### Evaluation / reliability flavored bullets
- Defined pass/fail criteria and review loops for AI outputs rather than relying on fluent responses as proof of correctness.
- Used evaluation-minded development practices to identify context drift, tool misuse, and silent failure modes.
- Treated AI outputs as production artifacts requiring verification, traceability, and measurable quality.

### Context / retrieval flavored bullets
- Structured operational documentation and knowledge surfaces so agents could retrieve relevant context on demand.
- Separated always-loaded guidance from task-specific references to reduce context pollution and improve focus.
- Iterated on bootfiles, reference docs, and task-context boundaries to improve agent performance.

## Halo / Hermes-flavored project framing

### Short external framing
Built agent-oriented tooling and workflow systems spanning:
- structured memory and work tracking
- cron-driven briefings and reporting
- local and remote agent execution surfaces
- context-aware documentation and governance
- evaluation-minded AI workflow design

### Stronger bullet variants
- Built and maintained an AI workflow toolkit for structured memory, task orchestration, reporting, and cron-driven operational briefings.
- Developed agent execution surfaces for queued jobs, local orchestration, and reviewable output flows.
- Designed governance and context patterns for AI-assisted work, including bootfile simplification, documentation structure, and task-specific context loading.
- Applied systems thinking to AI workflows by focusing on reliability, inspectability, and human review boundaries.

## Application tailoring rule

For each application, tune the language toward one of these four emphasis modes:

### Mode A — Applied AI Engineer
Emphasize:
- shipping systems
- reliability
- orchestration
- backend / APIs
- production judgment

### Mode B — Evaluation / Reliability role
Emphasize:
- eval harnesses
- correctness
- failure diagnosis
- guardrails
- observability

### Mode C — Context / Prompt / Retrieval role
Emphasize:
- specification clarity
- prompt strategy
- retrieval-aware design
- context packaging
- documentation architecture

### Mode D — Technical Product / hybrid role
Emphasize:
- decomposition
- workflow design
- trust boundaries
- prioritization by risk and value
- translating ambiguous needs into operational systems

## Simple daily operating loop for the search

1. Run 5–8 saved searches using the query pack
2. Save high-signal roles into buckets:
   - applied AI engineer
   - evaluation / reliability
   - prompt-context-retrieval
   - product / hybrid
3. For each saved role, note repeated capability language
4. Update CV bullets only when a phrase recurs across multiple roles
5. Build portfolio pieces that strengthen the most repeated capability clusters

## Final guidance

Do not market as "someone who uses AI."
Market as someone who can:
- specify AI work clearly
- make agents reliable
- build evaluation and review into workflows
- structure context so systems can perform
- place the right guardrails around autonomy
- reason about when the automation is worth it
